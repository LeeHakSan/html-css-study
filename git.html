Chapter 01

Non-fast-forward 에러란?

git push 할 때 가장 자주 만나는 거절 메시지입니다. 한 줄로 말하면:

"내가 보내려는 커밋이 원격의 최신 상태 위에 곧바로 얹히지 못하는 상황"

원격에 내가 모르는 커밋이 있어서, 그냥 push하면 그 커밋이 사라지기 때문에 Git이 안전장치로 막아주는 것입니다.

💡
핵심 포인트 거절은 에러가 아니라 보호 장치입니다. Git은 다른 사람의 작업이 조용히 사라지는 걸 막기 위해 일부러 push를 실패시킵니다.
Chapter 02

Fast-forward 먼저 이해하기

Fast-forward(빨리감기)의 어원

옛날 비디오테이프의 빨리감기 버튼을 떠올리면 됩니다. 테이프를 중간에 끊거나 바꾸지 않고, 현재 위치에서 같은 선 위로만 앞으로 쭉 감는 동작입니다.

🎞️
비디오테이프 비유 Rewind는 뒤로 감기, Fast-forward는 앞으로 감기. 핵심은 뒤로 가거나 다른 경로로 새지 않고, 같은 선 위에서 앞으로만 이동한다는 것.

Git에서의 Fast-forward

Git에서 브랜치는 커밋들이 한 줄로 이어진 "테이프"이고, 브랜치 이름(main, feature/login 등)은 현재 위치를 가리키는 포인터입니다.

커밋 C까지 진행된 상태에서 D, E가 같은 선 위에 추가되었다고 가정해봅시다. main 포인터가 C에서 E로 이동하는 과정을 Before / After로 보면:

Before · 이동 전
커밋:
A
──
B
──
C
──
D
──
E
▲
main
After · 이동 후
커밋:
A
──
B
──
C
──
D
──
E
▲
main

main을 C에서 E로 옮기는 건 새로운 경로를 만드는 게 아니라, 이미 있는 선 위에서 포인터만 앞으로 미는 것입니다. 이게 fast-forward입니다.

Chapter 03

Non-fast-forward가 발생하는 상황

다음과 같은 상황을 봅시다. 팀원이 먼저 X를 push한 사이, 내가 로컬에서 C, D 커밋을 만들었습니다.

원격 main: A ── B ── X (팀원이 X를 먼저 push) 내 로컬: A ── B ── C ── D (내가 C, D를 만드는 동안)

내 로컬은 B에서 분기해서 C, D를 만들었는데, 그 사이 팀원이 X를 원격에 먼저 올렸습니다. 이제 원격에는 내가 모르는 커밋 X가 있는 상태입니다.

이 상태에서 git push를 하면:

terminal — push 거절 메시지
$ git push ! [rejected] main -> main (non-fast-forward) error: failed to push some refs to '...' hint: Updates were rejected because the remote contains work that you do not have locally.
🛑
왜 거절할까? 그냥 내 커밋(C, D)으로 밀어버리면 팀원의 X 커밋이 사라지기 때문입니다. Git이 "잠깐, 너 원격에 새로 생긴 거 모르지?" 하고 막아주는 것.
Chapter 04

올바른 해결 순서

원격의 새 커밋을 받아와서 내 로컬에 합친 뒤, 다시 push하면 됩니다.

1

원격의 변경사항 받아오기

git pull 은 내부적으로 fetch + merge 를 수행합니다. 원격의 X가 내 로컬 히스토리에 합쳐집니다.

2

충돌이 있으면 해결

두 쪽이 같은 파일을 건드렸다면 충돌 마커를 열어서 수동으로 정리한 뒤 add, commit 합니다.

3

다시 push

이제 내 로컬 tip이 원격 tip 위에 있으므로 fast-forward가 가능해져서 push가 성공합니다.

terminal — 정석 해결 흐름
$ git pull origin main # 원격의 X를 받아와서 내 로컬에 병합 # → 자동 병합 커밋 생기거나, # 충돌 나면 해결 후 commit
$ git push # 이제 fast-forward 가능 → 성공

병합 후 히스토리

A
──
B
──
X
─────────
M
(M은 병합 커밋)
╲
╱
C
─────────
D
🔀
rebase 대안 히스토리를 일직선으로 유지하고 싶다면 git pull --rebase 또는 git rebase origin/main 으로 내 커밋을 원격 tip 위로 옮겨 붙일 수 있습니다. 단, 이 경우 push 할 때 --force-with-lease가 필요합니다.
Chapter 05

절대 하지 말아야 할 것: push --force

--force 옵션은 "원격의 X를 무시하고 내 C, D로 덮어써!" 라는 뜻입니다. push는 성공하지만 대가가 큽니다.

☠️
force push가 남기는 흔적 팀원의 X 커밋이 영구 소실됩니다. 팀원은 다음 pull 때 "내 커밋 어디 갔지?" 하며 혼란에 빠지고, 팀에서 신뢰를 잃는 지름길입니다.

안전한 대안: --force-with-lease

강제로 덮어써야만 하는 상황(예: 내 브랜치에서 rebase 후 push)이라면 --force-with-lease를 씁니다.

terminal — 조건부 강제 push
$ git push --force-with-lease
"내가 마지막으로 본 원격 상태가 그대로일 때만 force" 라는 조건부 강제 push. 내가 fetch한 이후 누군가 원격에 새 커밋을 올렸다면 거절됩니다.
Chapter 06

비유로 정리

상황 비유
Fast-forward 같은 도로 위에서 액셀을 밟는 것 (한 방향 직진)
Non-fast-forward 갈림길에서 다른 길로 점프하는 것 (불가능 — 먼저 합류부터)
git pull 갈라진 두 길을 다시 하나로 합치는 합류 공사
git push --force 표지판을 몰래 바꿔서 팀원 길을 지워버리기 (금지!)
--force-with-lease "내가 본 이후 바뀐 거 없지?" 확인하고 덮어쓰기 (조건부)
한 줄 요약
Fast-forward = 브랜치 포인터를 이미 존재하는 일직선 경로 위에서 앞으로 이동시키는 가장 단순·안전한 업데이트.

Non-fast-forward = 그게 불가능한 상황(원격에 내가 모르는 커밋이 있음). pull 먼저, 그다음 push.

체크리스트 — 에러 만났을 때

git fetch 로 원격 상태 먼저 확인
git pull origin <브랜치> 로 원격 커밋 받아오기
충돌 나면 충돌 마커(<<<<<<<, =======, >>>>>>>) 정리 후 git add, git commit
다시 git push
절대 --force 옵션 먼저 쓰지 않기 (공유 브랜치라면 더더욱)

관련 개념

📌
함께 알아두면 좋은 것
  • git pull = git fetch + git merge
  • git rebase — 내 커밋을 원격 tip 위로 옮겨 붙여 히스토리를 일직선으로 만드는 대안 (push 시 force-with-lease 필요)
  • Protected branch — GitHub에서 main 같은 중요 브랜치는 force push 자체를 막아두는 것이 실무 표준
