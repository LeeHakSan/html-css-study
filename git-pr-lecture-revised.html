Chapter 01

브랜치(Branch)란 무엇인가?

브랜치는 코드의 독립된 작업 공간입니다. 원본 코드를 건드리지 않으면서 새 기능을 개발하거나 버그를 수정할 수 있는 분리된 줄기입니다.

💡
비유로 이해하기 책의 원고를 복사해서 편집하는 것처럼 — 원본은 그대로 두고 복사본에서 마음껏 수정합니다. 완성되면 원본에 합치면 됩니다.
main ●──────────────────────────────●──● (배포용 브랜치) ╲ ╱ feature ●────●────●────● (내 작업 브랜치) c1 c2 c3 c4

main 브랜치는 항상 배포 가능한 안정적인 코드를 유지합니다. 새로운 작업은 feature 브랜치에서 독립적으로 진행합니다.

🛡️ 안전성

실험적인 변경이 main 브랜치를 망가뜨리지 않습니다. 언제든 원본으로 돌아올 수 있습니다.

🤝 협업

팀원 A는 로그인 기능, 팀원 B는 검색 기능을 동시에 따로따로 개발할 수 있습니다.

📋 이력 관리

각 기능의 작업 내역이 브랜치 단위로 명확하게 구분되어 추적이 쉽습니다.

🔁 롤백

문제가 생기면 해당 브랜치만 폐기하면 됩니다. main은 절대 영향받지 않습니다.

Chapter 02

브랜치 생성과 전환 실습

실제로 브랜치를 만들고 이동하는 명령어를 배웁니다. 브랜치 이름은 무엇을 하는 브랜치인지 명확하게 짓는 게 중요합니다.

브랜치 기본 명령어
# 현재 존재하는 브랜치 목록 확인 $ git branch * main develop
# 새 브랜치 만들기 $ git branch feature/add-login
# 브랜치로 이동 $ git switch feature/add-login Switched to branch 'feature/add-login'
# 브랜치 만들면서 바로 이동 (위 두 명령을 한 번에 처리) $ git switch -c feature/add-signup Switched to a new branch 'feature/add-signup'
✅
좋은 브랜치 네이밍 컨벤션 feature/기능이름 — 새 기능 추가
fix/버그이름 — 버그 수정
hotfix/긴급수정 — 운영 중 긴급 패치
refactor/대상 — 코드 개선 (기능 변경 없음)
브랜치에서 작업하고 커밋하기
# 파일 수정 후 스테이징 $ git add src/login.js
# 명확한 메시지로 커밋 $ git commit -m "feat: 로그인 폼 UI 추가" [feature/add-login 3a1b2c0] feat: 로그인 폼 UI 추가
$ git commit -m "feat: 이메일 유효성 검사 로직 추가" $ git commit -m "feat: 로그인 API 연동 완료"
⚠️
커밋 메시지가 중요한 이유 나중에 PR을 만들 때 커밋 메시지들이 변경 이력이 됩니다. "수정", "업데이트" 같은 모호한 메시지는 리뷰어와 미래의 나를 괴롭힙니다.
Chapter 03

PR(Pull Request)이란 무엇인가?

PR은 "내가 만든 브랜치를 main 브랜치에 합쳐줘"라고 팀에게 요청하는 공식 절차입니다. 단순히 코드를 합치는 게 아니라, 코드 리뷰 → 토론 → 검증 → 병합의 전체 프로세스를 담은 협업 도구입니다.

브랜치 작업
feature/*
→
Push
origin에 올리기
→
PR 생성
GitHub에서
→
코드 리뷰
팀원 검토
→
Merge
main에 병합

왜 이렇게 복잡한 절차를 거칠까요? 단 한 줄의 잘못된 코드가 운영 서버를 다운시킬 수 있기 때문입니다. PR은 그 코드가 main에 들어가기 전에 반드시 누군가가 확인하도록 강제하는 안전망입니다.

Open
[feat] 로그인 기능 추가 #42
from feature/add-login
into main
author @jungmin-dev
commits 3 commits
## 변경 사항
- 이메일/비밀번호 로그인 폼 UI 추가
- 이메일 형식 유효성 검사 로직 구현
- /api/auth/login 엔드포인트 연동

## 테스트 방법
1. 로컬에서 npm run dev 실행
2. /login 페이지 접속
3. 올바른/잘못된 이메일로 각각 테스트
✅ 로컬 테스트 완료
✅ 코드 스타일 가이드 준수
✅ 관련 문서 업데이트
⬜ 코드 리뷰 승인 대기 중
💡
PR과 MR의 차이 GitHub에서는 PR (Pull Request), GitLab에서는 MR (Merge Request)라고 부릅니다. 이름만 다를 뿐 개념과 절차는 동일합니다.
Chapter 04

PR 생성부터 병합까지 실습

1

브랜치를 원격 저장소에 Push

로컬의 브랜치를 GitHub(원격)에 올려야 PR을 만들 수 있습니다.

브랜치 Push
# 처음 Push할 때 (upstream 설정 포함) $ git push -u origin feature/add-login Enumerating objects: 12, done. Branch 'feature/add-login' set up to track 'origin/feature/add-login'
# 이후 Push는 간단하게 $ git push
2

GitHub에서 PR 생성

Push 후 GitHub 저장소로 이동하면 노란 배너로 PR 생성 버튼이 나타납니다. Base 브랜치(병합 대상) → Compare 브랜치(내 브랜치) 방향을 확인하세요.

✅
좋은 PR 설명 작성법 ① 무엇을 변경했는가 (What)
② 왜 변경했는가 (Why)
③ 어떻게 테스트하나 (How to test)
④ 스크린샷 (UI 변경이라면 필수)
3

코드 리뷰

리뷰어는 변경된 파일(Files changed 탭)을 보며 코드를 검토합니다. 줄 단위로 댓글을 달 수 있고, 리뷰를 마무리할 때 세 가지 결론 중 하나를 선택합니다.

▼ 리뷰어가 선택하는 3가지 결론

✅ Approve

코드에 문제없음. 병합해도 좋다는 승인. 보통 1~2명의 승인이 필요합니다.

💬 Comment

승인/거부 없이 의견만 남김. "이 방법도 있어요" 같은 제안.

❌ Request Changes

수정이 필요한 부분 지적. 수정 완료 후 다시 리뷰를 요청해야 합니다.

🔄
Re-request Review — PR 작성자의 액션 위 3가지는 리뷰어가 내리는 결론이고, Re-request Review는 내가 리뷰어의 피드백을 반영해 수정·push한 뒤 "다시 봐주세요"라고 요청하는 작성자 측 기능입니다. PR 페이지의 리뷰어 이름 옆 🔄 아이콘을 클릭하면 됩니다.
리뷰 피드백 반영 후 추가 Push
# 리뷰어 지적사항 수정 후 $ git add src/login.js $ git commit -m "fix: 리뷰 반영 - 비밀번호 최소 길이 검사 추가" $ git push # → push하면 GitHub PR에 새 커밋이 자동으로 추가됩니다 (별도 작업 불필요)
4

Merge (병합)

승인을 받으면 Merge 버튼이 활성화됩니다. 세 가지 방식 중 팀 규칙에 맞는 것을 선택하세요.

방식 특징 사용 상황
Merge Commit 모든 커밋 이력 유지, 병합 커밋 생성 히스토리를 모두 보존할 때
Squash & Merge 브랜치의 모든 커밋을 1개로 압축 main을 깔끔하게 유지할 때 (추천)
Rebase & Merge 커밋을 main 위에 재배치, 선형 히스토리 깔끔한 선형 이력이 필요할 때
5

브랜치 정리 (선택사항)

Merge된 브랜치는 더 이상 필요 없습니다. GitHub에서 자동으로 삭제 버튼이 나타나며, 로컬에서도 정리합니다.

Merge 후 정리
# main으로 이동 후 최신 코드 받기 $ git switch main $ git pull origin main
# 로컬 브랜치 삭제 $ git branch -d feature/add-login Deleted branch feature/add-login
# 원격 브랜치도 삭제 (GitHub 버튼 또는 명령어) $ git push origin --delete feature/add-login
Chapter 05

전체 흐름 정리 & 실전 팁

지금까지 배운 내용을 하나의 완전한 흐름으로 정리합니다.

처음부터 끝까지 전체 명령어 흐름
──────── 1. 브랜치 생성 ──────── $ git switch -c feature/user-profile
──────── 2. 작업 & 커밋 ──────── $ git add . $ git commit -m "feat: 유저 프로필 페이지 추가"
──────── 3. 원격에 Push ──────── $ git push -u origin feature/user-profile
──────── 4. GitHub에서 PR 생성 → 리뷰 → Merge (GUI) ────────
──────── 5. 로컬 정리 ──────── $ git switch main && git pull $ git branch -d feature/user-profile

실전에서 자주 하는 실수들

🚫
main 브랜치에 직접 Push하지 마세요! 팀 프로젝트에서 main에 직접 push하는 것은 팀의 코드 리뷰 절차를 우회하는 행위입니다. 항상 브랜치를 만들고 PR을 통해 병합하세요.
⚠️
오래된 브랜치를 방치하지 마세요 기능 개발 중에도 git pull origin main을 정기적으로 실행해서 최신 main을 내 브랜치에 반영하세요. 나중에 merge conflict가 훨씬 적어집니다.
※ feature 브랜치에 있을 때 이 명령은 원격 main을 내 브랜치에 머지합니다. 팀이 rebase 워크플로우를 쓴다면 git fetch origin && git rebase origin/main을 대신 사용하세요.
💡
작은 PR이 좋은 PR입니다 500줄짜리 PR보다 50줄짜리 PR 10개가 훨씬 리뷰하기 쉽습니다. 기능을 작게 나눠서 자주 PR을 만드는 습관이 팀 생산성을 높입니다.
한 줄 요약

브랜치는 안전한 작업 공간이고,
PR은 코드를 검증하고 합치는 공식 절차입니다.

이 두 가지가 팀 협업의 핵심입니다. 처음엔 번거로워 보여도, 이 과정 덕분에 팀 전체의 코드가 안전하게 유지됩니다.
